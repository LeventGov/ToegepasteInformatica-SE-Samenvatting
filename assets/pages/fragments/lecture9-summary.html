1. Wat is Cloud Computing?

Vroeger moesten bedrijven hun eigen servers kopen, koelen en beheren (On-Premises).

Definitie

Cloud computing is het leveren van computerdiensten (servers, opslag, databases, software) via het internet ("de cloud").

Het Nutsbedrijf-model (Utility)

De cloud is als elektriciteit of water uit de kraan. Je bouwt geen eigen energiecentrale; je plugt in en betaalt achteraf voor wat je hebt verbruikt.

Voordelen

  • Geen CapEx (Capital Expenditure): Geen grote investering vooraf in hardware
  • OpEx (Operating Expenditure): Je betaalt maandelijks (operationele kosten)
  • Elasticiteit: Je kunt binnen enkele minuten opschalen van 1 naar 1000 servers

2. Service Modellen (HET Examenonderdeel!)

Het belangrijkste concept van deze les is de verdeling van verantwoordelijkheid. Wie beheert wat? Dit wordt vaak uitgelegd met de "Pizza as a Service" analogie of de "Stack".

A. On-Premises (Alles zelf doen)

  • Jij beheert alles: Van de netwerkkabels en de airco tot de applicatie en de data
  • Analogie: Thuis pizza maken (je koopt de oven, het deeg, de kaas, bakt het en eet het op)

B. IaaS (Infrastructure as a Service)

  • Wat huur je: De virtuele hardware (Servers/VM's, Opslag, Netwerk)
  • Wat beheer jij: Het Besturingssysteem (Windows/Linux), Middleware, Data en Applicaties
  • Provider regelt: Virtualisatie, Servers, Opslag, Netwerk
  • Voorbeelden: Amazon EC2, DigitalOcean Droplets, Azure VM
  • Analogie: "Take & Bake" (Je huurt de keuken/oven, maar brengt je eigen pizza mee)

C. PaaS (Platform as a Service)

  • Wat huur je: Een omgeving om code te draaien (Runtime)
  • Wat beheer jij: Alleen je Applicatie en je Data
  • Provider regelt: Het OS, patches, updates, middleware en de hardware
  • Voorbeelden: Heroku, Google App Engine, Azure App Service
  • Analogie: Pizza laten bezorgen (Je kiest de pizza, maar iemand anders bakt hem en bezorgt hem. Jij hoeft alleen de tafel te dekken)

D. SaaS (Software as a Service)

  • Wat huur je: De volledige applicatie
  • Wat beheer jij: Niets (behalve misschien gebruikersinstellingen)
  • Voorbeelden: Gmail, Microsoft Office 365, Dropbox
  • Analogie: Uit eten gaan (Je gaat zitten, eet en betaalt. Je hebt geen idee hoe de keuken eruitziet)

3. Deployment Modellen

Waar staat de hardware fysiek en wie mag erop?

  • Public Cloud: Alles staat bij de provider (AWS, Azure). Hardware wordt gedeeld met andere klanten (Multi-tenancy). Goedkoopst en meest schaalbaar
  • Private Cloud: Een cloud-omgeving die exclusief voor één bedrijf is. Kan in je eigen datacenter staan of gehuurd zijn. Veilig, maar duur
  • Hybrid Cloud: Een combinatie. Bijvoorbeeld: Gevoelige klantdata staat op een Private Cloud (veilig), maar de website draait in de Public Cloud (schaalbaar)
  • Multi Cloud: Je gebruikt meerdere providers tegelijk (bijv. AWS voor opslag en Azure voor AI) om "Vendor Lock-in" te voorkomen

4. Virtualisatie & Resource Pooling

De motor achter de cloud is virtualisatie (zoals gezien in Lecture 4).

Resource Pooling

De provider heeft enorme hallen vol servers. Al deze CPU-kracht en RAM wordt op één grote hoop gegooid ("Pool").

Multi-tenancy

Als jij een VM huurt, weet je niet dat jouw VM fysiek op dezelfde server draait als de VM van je concurrent. De Hypervisor zorgt voor strikte isolatie.

Analogie: Een flatgebouw. Iedereen heeft zijn eigen sleutel en appartement (isolatie), maar jullie delen allemaal dezelfde waterleiding en fundering (resources).

5. Schaalbaarheid (Scaling)

Er zijn twee manieren om te groeien als je app populair wordt:

Vertical Scaling (Scale Up)

  • Wat: Je maakt de server groter (meer CPU, meer RAM)
  • Nadeel: Er is een limiet (fysieke hardware houdt ergens op) en je moet vaak herstarten (downtime)

Horizontal Scaling (Scale Out) - De Cloud Manier

  • Wat: Je zet er meer servers naast
  • Voordeel: Theoretisch oneindig schaalbaar. Als 1 server vol is, start de cloud automatisch een tweede, derde, etc.
  • Tool: Load Balancer (verdeelt het verkeer over de servers)

6. Cloud Management (IAM)

Wie mag wat doen in jouw cloud-omgeving?

  • Authentication (AuthN): Bewijzen wie je bent (Inloggen/Paspoort)
  • Authorization (AuthZ): Bepalen wat je mag (Rechten/Visum)
  • RBAC (Role-Based Access Control): Je geeft rechten niet per persoon, maar per rol
    • Voorbeeld: Piet is "Developer" → mag VM's starten. Klaas is "Finance" → mag facturen zien

7. SLA & Billing

SLA (Service Level Agreement)

Het contract waarin de provider belooft hoe betrouwbaar ze zijn.

  • Uptime: Vaak uitgedrukt in nines (bijv. 99.99%)

Billing Models

  • Pay-as-you-go: Betalen per seconde/GB
  • Spot Pricing: Je biedt op ongebruikte capaciteit. Heel goedkoop, maar de provider kan de server elk moment afpakken als ze hem zelf nodig hebben

8. Security: Shared Responsibility Model (Cruciaal!)

Veel mensen denken: "Ik zit in de cloud, dus Microsoft regelt de beveiliging." FOUT.

  • Cloud Provider: Verantwoordelijk voor de beveiliging VAN de cloud
    • (Fysieke datacenters, kabels, hypervisor, netwerk)
  • Klant (Jij): Verantwoordelijk voor de beveiliging IN de cloud
    • (Jouw data, jouw wachtwoorden, updates van jouw OS, firewall instellingen)

Belangrijk: Als jij je AWS-wachtwoord op GitHub zet en gehackt wordt, is dat JOUW schuld, niet die van Amazon.

Analogie voor IaaS/PaaS/SaaS: Wonen

  • On-Premises: Je bouwt zelf een huis, legt de leidingen, regelt gas/water/licht en het onderhoud
  • IaaS: Je huurt een kaal huis. De huisbaas regelt dat het dak niet lekt (Infrastructuur). Jij moet zelf meubels kopen, schilderen en schoonmaken (OS/App)
  • PaaS: Je huurt een gemeubileerd hotel. Het bed staat er (Runtime), je hoeft alleen je kleren (Code) in de kast te leggen
  • SaaS: Je woont in een all-inclusive resort. Je hoeft niet te koken, niet schoon te maken, alleen maar te genieten (Gebruiken)
